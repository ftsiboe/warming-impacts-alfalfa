# Proposal — Replace Hard-Coded Manuscript Numbers with an `article_objects.json` Layer

**Goal.** Every number quoted inline in the alfalfa manuscript (yield-response
percentages, warming-scenario impacts, state/county figures, summary statistics,
correlations) should be *computed once from the analysis outputs* and read into the
prose from a single JSON file — exactly the pattern used in `proj-prf-redesign`. No
result is ever typed by hand into a section `.Rmd`. This directly closes review
comments C1, C2, C5, C8, C9, C10 (all "pull from Spatial Impact Yield / update
needed") and prevents future drift between the prose, the tables, and Figure 5.

Nothing here is implemented yet — this is the design for approval.

---

## 1. How `proj-prf-redesign` does it (the pattern we copy)

Four moving parts, all under `scripts/` + `narrative/`:

1. **`200_article_helpers.R`** — defines repo-root-relative paths (`DATA`, `OUTPUT`,
   `EXHIBITS`, `NARRATIVE`, `OBJECTS_JSON`), analysis constants, and formatting
   helpers (`fmt_pct()`, `fmt_num()`, en-dash).
2. **`201_article_objects.R`** — reads the exhibit CSVs/summary objects, computes
   *every* inline number, assembles them into one nested list `objs`, and writes
   `narrative/article_objects.json` via `jsonlite::write_json(..., auto_unbox = TRUE)`.
   Tables are stored as data frames inside `objs` too.
3. **`300_render_article.R`** — `rmarkdown::render()` of the master `.Rmd` with
   `knit_root_dir = narrative/` so child and object paths resolve.
4. **`run_article.R`** — sources 200 → 201 → 300 in order. The master `.Rmd` opens with
   `objs <- jsonlite::fromJSON("article_objects.json")` and pulls in `sections/*.Rmd`
   as knitr children. Sections reference values inline as `` `r objs$act$opt_window` ``
   and emit tables with `knitr::kable(objs$act$tab)`.

Key property: the article layer **only consumes** exhibits produced by the analysis
pipeline; it never re-runs the model.

## 2. Proposed structure for this project

Add to `scripts/`:

```
scripts/
  200_article_helpers.R     # paths, constants, fmt_pct/fmt_num helpers
  201_article_objects.R     # build narrative/article_objects.json
  300_render_article.R      # knit the master Rmd -> narrative .docx
  run_article.R             # source 200 -> 201 -> 300
```

Add to `narrative/`:

```
narrative/
  article_warming_effects_alfalfa.Rmd   # master; reads objs, stitches sections/
  article_objects.json                  # GENERATED by 201 (do not edit by hand)
  reference.docx                        # Word style reference for rendering
  references.bib                        # bibliography (also resolves C6/C7)
  sections/                             # existing split files, edited to use objs$
```

This slots cleanly beside the pipeline already in place (`000`–`200` +
`100_...exhibits.R`). The `100` exhibits script is the producer of the CSVs the
`201` article builder consumes.

## 7. Guardrails (so the JSON stays trustworthy)

- **Single source of truth:** a value appears in `201_article_objects.R` exactly once;
  sections never recompute it. If two sentences cite the same number they read the
  same field.
- **Fail loud:** `201` should `stop()` if any expected value is `NA`/missing, so a
  broken upstream exhibit cannot silently ship a blank into the manuscript (this is
  what the "XXX%" placeholders are today).
- **Rounding lives in helpers:** prose calls `fmt_pct()`/`fmt_num()`; raw numbers stay
  full-precision in JSON so display precision is changed in one place.
- **Regeneration is one command:** `source("scripts/run_article.R")` rebuilds JSON and
  knits. A stale number is impossible if the runner is used.
- **`article_objects.json` is a build artifact** — regenerated, not hand-edited; note
  this at the top of the file and in the README.
